**2. 录音格式兼容性处理链**

不同浏览器输出不同音频格式（Desktop Chrome → WAV，iOS Safari → AAC/MP4，微信 → WebM），设计统一的服务端转码流水线：`前端检测 MIME → 上传 → 后端判断格式 → ffmpeg 按需转码 16kHz Mono WAV → ASR`

**3. IndexedDB 离线暂存 + 恢复机制**

用户在录音/对话中遇到网络中断或 Token 过期时，已录音频需暂存 IndexedDB，场景恢复后自动重试上传。设计前端暂存策略和服务端幂等上传接口。

**4. LLM 对话与评分分离调用**

对话 Prompt 仅返回角色文本回复（低延迟，<3s），评分在对话结束后独立调用（Temperature=0.1-0.3，传入完整 5 轮历史，可接受较长耗时）。避免单次 LLM 调用同时做对话+评分导致延迟不可控。

**5. 多态关联处理**

`study_sessions.reference_id` 同时指向 `practice_records` 或 `conversation_sessions`，无法建 DB 级 FK。设计应用层一致性保证方案，给出具体实现策略。

**6. 定时任务设计**

语音数据 168h 精确删除：每小时轮询 `practice_records.created_at ≤ NOW() - 168h AND audio_deleted_at IS NULL` 的记录，物理删除文件 + 更新 `audio_deleted_at` + 写入 audit_logs。
