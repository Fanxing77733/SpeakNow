## 3. API 契约对齐检查结果

### 3.1 baseURL 与请求路径检查

| 检查项 | 结果 |
|--------|------|
| FE `src/api/client.ts` baseURL | `/api/v1` -- **正确** |
| 所有 API 函数的 URL 格式 | 相对路径（如 `/auth/login`），从 baseURL 继承前缀 `/api/v1/` -- **正确** |
| 实际请求路径示例 | `POST /api/v1/auth/login`, `GET /api/v1/assessment/questions?type=fixed`, `POST /api/v1/eval/pronounce`, `POST /api/v1/chat/session/start`, `GET /api/v1/progress/summary` -- **全部正确** |

### 3.2 关键字段对齐问题

| 编号 | 模块 | 问题 | 严重程度 | 详情 |
|------|------|------|----------|------|
| **FIND-001** | 情景对话 | `POST /api/v1/chat/message` 字段不匹配 | **P0 阻塞** | 后端返回 `MessageVO`（round/role/content/createdAt），前端期望 `AIMessageResponse`（round/userText/aiText/sessionId/totalRounds）。`response.userText` 将为 undefined，**用户消息气泡内容为空**，对话页核心功能缺陷 |
| **FIND-002** | 情景对话 | `createdAt` 序列化格式 | **P1 高** | 后端 `LocalDateTime` 若无 Jackson 配置会序列化为数组 `[2026,6,26,10,30,0]`，前端 `new Date()` 解析失败。需确认 `spring.jackson.serialization.write-dates-as-timestamps=false` 已配置 |
| **FIND-003** | 测评 | `QuestionVO` 的 `options` 字段结构 | **P1 高** | API 规范定义 `options: { A: "...", B: "...", C: "...", D: "..." }`（对象），但前端类型定义为 `optionsJson: string`（JSON 字符串）。须确认后端实际序列化格式 |
| **FIND-004** | 测评 | `AnswerItem.selectedKey` vs `selectedOption` | **P1 高** | 前端发送 `{ questionId, selectedKey }`，API 规范要求 `{ questionId, selectedOption }`。字段名不一致导致后端无法正确接收用户答案 |
| **FIND-005** | 测评 | `AssessmentResult.resultLevel` vs API `level` | **P2 中** | 前端类型使用 `resultLevel`，API 规范及测试用例一致使用 `level`。若后端返回 `level`，前端取值失败 |
| **FIND-006** | 发音评测 | `wordResults` vs `wordDetails` | **P1 高** | 前端类型 `PronounceEvalResult.wordResults`，CT-PRACTICE-006 规定后端返回 `wordDetails`。字段名不匹配导致逐词颜色渲染失败 |
| **FIND-007** | 全局 | `Result<T>` 缺少 `timestamp` | P2 低 | 前端 `types/api.ts` 中 `Result<T>` 仅有 `code/message/data`，缺少后端始终返回的 `timestamp` 字段。当前 `request()` 仅解构 `data` 故无影响 |
| **FIND-008** | 用户中心 | `User.createdAt` 类型 | P2 低 | 前端定义为 `string`，须确认后端序列化为 ISO 8601 字符串而非数组 |

### 3.3 已验证对齐的字段

## 5. 已知问题与风险

### 5.1 阻塞问题（P0 — 必须修复才能进入 V2.0）

| 编号 | 问题 | 影响范围 | 建议修复方案 |
|------|------|---------|-------------|
| FIND-001 | 情景对话 `POST /chat/message` 字段不匹配 | 对话页核心功能完全不可用 | 方案 A（推荐）：后端新增 `MessageWithSessionVO` 返回 round/userText/aiText/sessionId/totalRounds 五字段。方案 B：前端适配 `MessageVO`，从 store 自行记录 userText。 |

### 5.2 高优先级问题（P1 — 影响核心体验）

| 编号 | 问题 | 影响范围 | 建议修复方案 |
|------|------|---------|-------------|
| FIND-002 | `createdAt` 序列化格式 | 所有模块日期展示 | 在 `application.yml` 配置 `spring.jackson.serialization.write-dates-as-timestamps=false` + `spring.jackson.date-format=yyyy-MM-dd'T'HH:mm:ss`，或在 Entity 字段加 `@JsonFormat(pattern = "yyyy-MM-dd'T'HH:mm:ss")`。建议统一使用 `Instant` 替代 `LocalDateTime` |
| FIND-003 | 测评题目 `options` 字段结构 | 测评答题页 4 个选项按钮渲染 | 统一：后端返回 `options: { A: "...", B: "...", C: "...", D: "..." }` 对象，前端类型改为 `options: Record<string, string>` 并相应调整解析逻辑 |
| FIND-004 | 答案提交 `selectedKey` vs `selectedOption` | 提交测评答案后端无法正确评分 | 统一字段名为 `selectedOption`（对应 API 规范），修改前端 `AnswerItem` 和测评 store |
| FIND-006 | 发音评测 `wordResults` vs `wordDetails` | 发音评测结果页逐词颜色渲染失败 | 统一字段名为 `wordDetails`（对应 API 规范 CT-PRACTICE-006/007），修改前端 `PronounceEvalResult` 接口 |

### 5.3 中低优先级问题（P2 — 不影响核心流程）

| 编号 | 问题 | 说明 |
|------|------|------|

| FIND-007 | `Result<T>` 缺少 `timestamp` | 当前不影响功能，后续如需展示时间需补充 |
| FIND-008 | `User.createdAt` 字符串格式依赖 | 须确认后端序列化正确 |
| TODO-001 | 录音 >60s 截断未展示用户提示 | useRecorder `stopRecording` 进行截断但无 Toast |
| TODO-002 | 录音上传前缺少 >5MB 大小校验 | 超大录音将未经前端拦截直接发送至后端 |
